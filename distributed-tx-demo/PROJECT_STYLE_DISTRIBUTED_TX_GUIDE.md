# 项目里真正使用的分布式事务方案

这份文档只讲一件事：

> **如果你在真实项目里，没有上 XA / Seata / TCC 框架，而是采用“本地事务 + 状态机 + 幂等 + 定时补偿/对账”的方式做一致性，这到底该怎么理解、怎么讲、怎么落地。**

先给结论：

这套方案本质上属于：

- 柔性事务
- 最终一致性方案
- 业务层分布式事务解法

它不是强一致分布式事务，但它是实际工作里非常常见的一种分布式事务实现方式。

---

## 1. 为什么项目里不用一个大事务解决

因为真实项目经常会跨：

- 本地数据库
- 外部银行 / 第三方渠道
- 异步任务
- 回执通知

这里最大的问题是：

> 外部系统根本不可能参加你本地数据库事务。

所以你不能指望用一个 `@Transactional` 把下面这些一步包住：

1. 插入本地指令单
2. 调银行接口
3. 等银行回执
4. 再更新本地状态

这就是为什么项目里真正的核心，不是“开一个全局事务”，而是设计一套**可收敛的一致性机制**。

---

## 2. 这套方案的 4 个核心组成

### 第一部分：本地事务

作用：

- 保证业务起点正确
- 保证“指令单 + 初始状态”原子落库

在这个项目里对应：

- `StateMachineDemoService.createInstruction(...)`

也就是说，先在本地事务里把：

- `requestNo`
- `amount`
- 初始状态 `WAIT_RECEIPT`
- `version`

一起落进去。

如果这一步失败，就一起失败，不会出现“半条业务数据”。

### 第二部分：状态机

作用：

- 约束状态只能按允许的方向流转
- 防止乱序回执把终态冲回去

在这个项目里对应：

- `StateMachineDemoService.applyReceipt(...)`
- `StateMachineDemoService.canTransit(...)`

比如：

- `WAIT_RECEIPT -> SUCCESS` 可以
- `WAIT_RECEIPT -> PROCESSING` 可以
- 但 `SUCCESS -> PROCESSING` 不可以

这就是为什么“成功回执先到，后面迟到的处理中回执”不会把数据改乱。

### 第三部分：幂等

作用：

- 防止同一笔业务被重复处理
- 防止回执重复到达导致重复生效

在这个项目里，幂等不是单独起一个复杂组件，而是落在这两个点上：

1. 业务唯一标识 `requestNo`
2. 状态更新必须基于当前状态和版本推进

也就是说，同一条记录一直围绕同一个 `requestNo` 更新，而不是每次回执来都新建一条业务。

### 第四部分：定时补偿 / 对账

作用：

- 处理长时间未终态的数据
- 处理系统中间崩溃、回执丢失、外部系统抖动

在这个项目里对应：

- `StateMachineDemoService.reconcileTimeoutInstructions(...)`
- `StateMachineDemoService.timeoutCompensationDemo(...)`

意思是：

- 如果状态长时间停留在 `WAIT_RECEIPT` 或 `PROCESSING`
- 平台不能永远等
- 就要靠定时任务扫描超时记录，再进行补偿或对账收敛

## 3. 用一句话解释这套方案

你可以直接这样说：

> **我们项目里没有直接上 XA / Seata 这种强一致分布式事务，而是采用本地事务保证起点正确，再结合状态机、幂等和定时补偿/对账，来实现最终一致性。**

这句话非常稳。

---

## 4. 它和 `@Transactional` 的边界怎么讲

要讲清楚：

### `@Transactional` 能保证的

- 单库内多表更新原子提交
- 单数据源回滚

### `@Transactional` 不能保证的

- 银行接口调用一定成功
- 回执一定立刻回来
- 第三方系统和你一起参加同一个数据库事务

所以项目里的真正做法是：

1. **先用本地事务保存业务单据和初始状态**
2. **再通过异步回执 / 定时补偿去推进最终结果**

## 6. 你要按什么顺序讲给面试官

最推荐你这样讲：

### 第一句：先定性

> 这个场景本质上是分布式事务 / 分布式一致性问题，但不是用强一致大事务解决的。

### 第二句：再给方案

> 我们采用的是本地事务 + 状态机 + 幂等 + 定时补偿/对账的柔性事务方案，保证最终一致性。

### 第三句：再解释为什么

> 因为外部银行 / 第三方渠道不可能参加我们数据库事务，所以不能靠一个 `@Transactional` 或 XA 把全链路包起来。

### 第四句：再讲控制点

> 本地事务保证指令和初始状态落库原子成功；状态机控制状态只能往前推进；幂等防止重复回执重复生效；补偿和对账保证长时间未终态的数据最终能收敛。

这四句讲清楚，基本就已经比很多“只会背 Seata”的回答更真实。

---

## 7. 你可以直接背的版本

> 我们项目里真正采用的不是 XA 或 Seata 这种强一致分布式事务，而是柔性事务方案。具体做法是：先在本地事务里把业务指令和初始状态落库，保证起点正确；然后外部系统的执行结果通过回执异步回来；回执落地时不是直接覆盖状态，而是通过状态机和版本控制做条件更新，防止乱序和重复处理；对于长时间未终态的数据，再通过定时补偿和对账任务去收敛最终状态。所以本质上是“本地事务保证起点，状态机和幂等保证过程，补偿和对账保证最终一致”。

---

## 8. 这套方案到底算不算分布式事务

最准确的说法是：

- 它不是强一致分布式事务
- 但它是分布式事务问题下的一种**业务层柔性事务解法**

所以面试里不要说成：

- “我们没做分布式事务”

更好的说法是：

> **我们没有采用强一致事务框架，而是采用业务层最终一致性方案来解决分布式事务问题。**
