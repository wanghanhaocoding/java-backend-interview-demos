# MySQL 慢查询教学方法

这份文档不是单纯给你看 SQL，而是告诉你：

> **怎么带着问题去复现、怎么带着证据去排查、怎么带着对比去验证。**

配套 SQL：

- `sql/00_enable_slow_log_example.sql`
- `sql/01_init_data.sql`
- `sql/02_teaching_cases.sql`

---

## 一、推荐教学顺序

如果你是自己练，或者后面要给同事讲，建议按这个顺序走。

### 第 1 步：先确认环境

先看版本：

```sql
SELECT VERSION();
```

要点：

- MySQL 8.0.18+ 才支持 `EXPLAIN ANALYZE`
- 如果版本低，就用 `EXPLAIN`

### 第 2 步：先确认 slow log 生效

不要一上来就跑业务 SQL。

先执行：

```sql
SELECT SLEEP(0.3);
```

然后去看 slow log。

要讲清楚：

> 如果慢日志都没开，后面“排查慢查询”这件事本身就是空的。

### 第 3 步：先做“函数破坏索引”案例

这是最适合入门的慢查询场景，因为它非常典型，也很容易讲清楚。

要让学习者回答：

1. 为什么 `DATE(created_at)` 容易把索引效果用坏？
2. 为什么改成范围查询后更合理？
3. 优化证据是什么？

### 第 4 步：再做“缺联合索引”案例

这个更贴近真实项目。

要讲清楚：

> 索引不是“where 里有谁就给谁建一个”，而是要一起考虑 `WHERE + ORDER BY + LIMIT`。

### 第 5 步：再做 `LIKE '%xxx'` 和隐式类型转换

这两个坑非常像线上问题：

- 搜索功能写成 `%关键字%`
- Java 代码传错参数类型

它们都不是“SQL 报错”，而是“能跑，但慢，且容易被忽略”。

## 二、统一排查框架：5 步法

你以后遇到慢查询，建议都按这个顺序查。

### 第一步：先定性，是不是 SQL 慢

不要一上来就加索引。

先问：

- 接口慢，是不是数据库造成的？
- 应用日志里数据库耗时高不高？
- slow log 里有没有对应 SQL？

如果 slow log 根本没有，那可能是：

- 锁等待
- 网络抖动
- 应用线程池排队
- 下游服务慢
- 代码串行处理过多

### 第二步：看 slow log 的核心字段

至少盯住这 4 个：

1. `Query_time`
2. `Rows_examined`
3. `Rows_sent`
4. SQL 原文

尤其要培养这个直觉：

> `Rows_examined` 很大，但 `Rows_sent` 很小，通常说明扫描太多、过滤太晚、索引没用好。

### 第三步：看执行计划

优先：

```sql
EXPLAIN ANALYZE <SQL>;
```

不支持的话：

```sql
EXPLAIN <SQL>;
```

重点不是死记字段，而是看整体特征：

- 扫描行数是否明显偏大
- 是否走了不合适的索引
- 是否存在额外排序压力
- 估算行数和实际行数是否偏差很大

### 第四步：归类问题

最常见的慢查询基本都能归到这几类：

1. 没索引
2. 有索引，但 SQL 写法把索引用坏了
3. 单列索引不够，需要联合索引
4. 深分页
5. `LIKE '%xxx'`
6. 隐式类型转换
7. 返回列太多，回表过重

### 第五步：优化后一定回归验证

不要“加完索引”就算结束。

至少再做 3 件事：

1. 再跑一次 `EXPLAIN ANALYZE`
2. 再看一次 slow log
3. 再看一次真实耗时

## 三、每个案例应该怎么讲

### 案例 1：索引列上套函数

你要讲的核心不是“`DATE()` 不能用”，而是：

> 对索引列先做函数计算，会让优化器更难按原始索引范围查找。

让学习者重点观察：

- 扫描行数
- 优化前后 SQL 写法
- 优化前后执行计划对比

### 案例 2：缺联合索引

要讲清楚：

> `WHERE + ORDER BY + LIMIT` 经常要一起设计索引。

不要把注意力只放在 `WHERE status='PAID'` 上。

这条 SQL 真正想要的是：

- 先按 `status` 过滤
- 再按 `amount` 排序
- 再快速拿前 N 条

所以 `(status, amount)` 比单列索引更贴题。

### 案例 3：`LIKE '%xxx'`

要讲清楚：

> 前导 `%` 让前缀不确定，普通 B+ 树索引很难高效帮你定位。

所以：

- `LIKE 'VIP%'` 还有机会
- `LIKE '%VIP%'` 通常更容易变慢

### 案例 4：隐式类型转换

要讲清楚：

> 列是字符串，参数却按数字比较，这种写法很容易在真实项目里悄悄出问题。

这类坑非常像 Java 项目里“参数类型传错但还能跑”的问题。

---

## 四、建议你训练学习者回答的 3 个问题

每做完一个案例，都让对方回答：

1. 这条 SQL 为什么慢？
2. 问题属于“写法问题”还是“索引设计问题”？
3. 你拿什么证据证明它优化成功了？

如果这 3 个问题都能答出来，才算真正掌握，而不是只看热闹。

---

## 五、最核心的一句话

这套练习最想让你形成的不是“背答案”，而是这个习惯：

> **先从 slow log 发现问题，再用 `EXPLAIN ANALYZE` 找原因，最后用改 SQL / 调整索引的方式验证优化是否真的生效。**
