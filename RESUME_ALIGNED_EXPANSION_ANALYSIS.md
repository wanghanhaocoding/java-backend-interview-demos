## 一、先说结论

当前仓库已经不是“内容不够”，而是出现了两个层次：

### 1. 已经覆盖得比较好的层

这些项目已经能支撑你讲大部分通用后端高频题：

- `spring-core-demo`
- `spring-tx-demo`
- `mysql-lock-mvcc-demo`
- `redis-lock-demo`
- `mq-cache-idempotency-demo`
- `distributed-tx-demo`
- `jvm-stability-interview-demo`

这几块基本已经覆盖了你简历中明确写过的：

- Spring / MyBatis
- MySQL 事务 / MVCC / 锁 / 慢 SQL
- Redis 分布式锁 / 缓存一致性
- MQ 可靠性 / 幂等
- 分布式事务 / 状态机 / 补偿
- OOM / Full GC / 死锁排查

### 2. 还没有补到位的层

仓库目前最欠缺的，不是 ES、Netty、NIO 这类额外方向，而是你简历里最有价值的三条业务主线还没有被拆成“能直接拿去讲”的专门 demo：

- `ScheduleCenter` 深挖不够
- `AsyncJobCenter` 深挖不够
- `司库信息系统` 的回执 / 状态机 / 预算 / 归集场景还不够聚焦

所以后续扩展重点，应该是“把已有通用 demo 继续往你的真实业务上贴”，而不是继续横向摊大饼。

## 二、当前仓库里，哪些项目最值得保留和优先强化

### 1. `jvm-stability-interview-demo`

这是当前仓库里和简历贴合度最高的项目之一。

原因：

- 直接挂钩 `AsyncJobCenter`、`ScheduleCenter`、`司库信息系统`
- 正好对应你简历里写过的 `OOM / Full GC / 死锁`
- 也正好对应真实面试里“你是怎么发现、定位、修复”的追问方式

建议：

- 保持高优先级
- 后续可以继续补 `jps / jstack / jmap / jstat / arthas` 的工具链口语版
- 但这块不需要再另起大项目，更多是补文档和脚本

### 2. `redis-lock-demo`

这是当前仓库里最能补“面试追问弱点”的项目之一。

原因：

- 真实面试里明确追问了 Redis 原生锁怎么写、为什么要超时、为什么要持有者校验、为什么要 Lua 解锁
- 也和你的简历内容完全匹配
- 里面顺带把线程池、并发计数、缓存初始化竞态、慢查询实验室串起来了

建议：

- 保持高优先级
- 后续可以把“为什么 ScheduleCenter 会拆 `schedulerPool` 和 `workerPool`”单独再抽成一个更贴项目的话术文档

### 3. `mq-cache-idempotency-demo`

这个项目和你简历高度一致，而且覆盖了司库系统和异步任务中心的高频追问。

原因：

- 你简历里有 Kafka 可靠性 / 幂等 / 积压 / 一致性
- 司库系统里也有回执重复、接口幂等、缓存一致性
- 真实面试里也问到了“幂等为什么不能先查库”

建议：

- 保持高优先级
- 后续更适合往“银行回执 / 回单制作 / 预算结果回写”这些真实业务题材上靠

### 4. `distributed-tx-demo`

这个项目和你的业务背景是匹配的，但表达上还可以继续向“资金类真实场景”靠拢。

原因：

- 你简历里明确写了分布式事务、状态机、幂等、补偿
- 司库系统的网银回执和状态约束，本质上就是这一套
- 真实面试里也追问了跨库一致性、多数据源事务、外部系统回执

建议：

- 保持高优先级
- 后续重点不是继续扩 Saga/TCC 花样，而是把“网银指令 -> 回执 -> 状态推进 -> 补偿扫描”再做得更贴简历

### 5. `mysql-lock-mvcc-demo`

这是补数据库基本盘的核心项目。

原因：

- 你真实面试里在 MySQL 基础概念上有失分
- 简历里又明确写了索引、事务、MVCC、锁、分库分表
- 这类题不需要很花哨，但必须讲得稳

建议：

- 保持高优先级
- 后续更值得补“索引 / 存储引擎 / B+ 树 / 回表 / 覆盖索引 / 联合索引设计”文档，而不是再扩成新服务

### 6. `spring-core-demo` + `spring-tx-demo`

这两个项目很有必要保留，而且都是真实短板补位。

原因：

- 真实面试里明显暴露出 Spring 从零搭建、IoC、BeanDefinition、事务传播、事务边界这块答得不够结构化
- 这部分不是你的简历主战场，但属于基础题失分项

建议：

- 保持高优先级
- 但它们更像“补基本功”，不是你最强项目表达主线

## 三、当前仓库里，哪些项目方向有价值，但需要降级或改题材

### 1. `resilience-auth-demo`

这个项目有价值，但应该定义成“补短板项目”，不是主线项目。

原因：

- 登录认证、权限控制确实是你真实面试的弱点
- 但你的简历里没有把认证授权作为核心产出写出来
- 所以它适合用来补答题，不适合当成你简历主打项目去讲太久

建议定位：

- 保留
- 排在第二梯队
- 主用途是补“从零搭 Spring Boot + 登录鉴权 + RBAC”这一类面试题

### 2. `order-observability-pattern-demo`

这个项目的技术点没问题，但题材偏了。

问题不在“可观测性”或“设计模式”本身，而在场景选成了电商 checkout：

- 你的简历主线不是电商订单
- 你真实经历是司库、回执、预算、归集、票据
- 如果你拿这个项目做重点表达，会和简历背景出现题材漂移

建议定位：

- 不建议删
- 但建议后续改造成“司库回执 / 资金归集 / 日计划任务链路”的版本
- 也就是说，保留 trace、结构化日志、策略模式、责任链、模板方法这些技术点，但把业务外壳换掉

## 四、当前仓库最该新增的 6 个 demo

下面这些不是“可有可无”的扩展，而是最贴你简历、最容易被追问、最值得补齐的缺口。

### 1. `schedule-center-deep-dive-demo`

优先级：最高

建议覆盖：

- Redis ZSet 时间索引
- 按触发时间分片
- 滑动时间窗扫描
- 预取 + 本地队列削峰
- `schedulerPool` 与 `workerPool` 双线程池
- 多机扫描去重
- 同秒高峰任务打散

这会直接解决你真实面试里这些追问：

- ZSet 怎么扫
- 多节点一起扫怎么不重复
- 两个线程池分别干什么
- 同秒任务洪峰怎么处理
- 吞吐 400 到 1600 到底优化了哪些点

### 2. `async-job-center-deep-dive-demo`

优先级：最高

建议覆盖：

- server + worker 架构
- 三张核心表
- `order_time` 统一排序字段
- 失败重试 / 补偿 / 断点续做
- 行锁方案的问题
- Redis 分布式锁替代行锁
- 滚动分表 / 冷热拆分

这会直接对应你简历中最强的一段项目经历。

### 3. `treasury-receipt-state-machine-demo`

优先级：最高

建议覆盖：

- 网银指令发起
- 回执延迟 / 重复 / 乱序
- 幂等键
- 状态机流转约束
- 终态保护
- 版本控制
- 超时补偿 / 对账扫描

这是把 `distributed-tx-demo` 再往真实司库业务上推进一步。

### 4. `multi-datasource-transaction-demo`

优先级：高

建议覆盖：

- 单数据源事务
- 多数据源事务管理器分别管理
- 为什么不能混用事务管理器
- 一个业务同时写两个库为什么应该尽量避免
- 如果确实跨库，应该怎样退化为业务补偿 / 最终一致

这块价值非常高，因为真实面试里被连续追问，而且你目前仓库里还没有单独把这题讲透。

### 5. `distributed-id-demo`

优先级：高

建议覆盖：

- UUID、数据库自增、号段、雪花算法对比
- 为什么数据库更喜欢趋势递增 ID
- 无序主键为什么容易导致页分裂、插入抖动
- 如何做“分布式 + 趋势有序” ID
- ID 排序和 `create_time` 索引之间的关系

这是你真实面试中单独被追问过的一段，值得做成小而专的 demo。

### 6. `business-sql-case-demo`

优先级：高

建议覆盖：

- 首笔借款 / 首次任务 / 首次指令这类“首笔”判定
- 逾期 / 已结清 / 未结清未逾期 的状态区分
- 先确认字段含义，再下 SQL
- 复杂业务筛选如何拆成可验证步骤

如果题材能换成更贴近你的司库或异步任务业务，会比泛金融借款题更自然。
