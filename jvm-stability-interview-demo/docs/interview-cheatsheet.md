# JVM 稳定性问题面试速查

## 一、如果面试官问：你线上遇到过 OOM / Full GC / 死锁吗？

你可以先这样总答：

> 遇到过，类型不一样，排查路径也不一样。OOM 我更关注对象是不是被长期持有；Full GC 我更关注老年代为什么一直降不下来；死锁则要先拿线程 dump 看是不是互相等待锁。结合我做过的 AsyncJobCenter、ScheduleCenter 和司库系统，我分别准备了三个比较典型的案例。

## 二、三个案例的定位关键词

### 1. OOM
- `heap dump`
- `MAT`
- `长期持有对象`
- `无上限集合`

### 2. Full GC
- `Old 区高`
- `Full GC 频繁`
- `批量预取过大`
- `背压不足`

### 3. 死锁
- `线程 dump`
- `BLOCKED`
- `waiting to lock`
- `锁顺序不一致`

## 三、一个万能收尾句

> 我现在看这类问题，都会尽量区分“现象”和“根因”。比如 OOM 不是简单堆太小，Full GC 不是简单 GC 参数问题，死锁也不是简单线程多，而是对象生命周期、批处理模型、锁设计这些更底层的东西出了问题。排查时我会先看监控和现场证据，再回到代码和业务链路去收敛根因。
