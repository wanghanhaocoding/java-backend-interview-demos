# 死锁案例：任务状态更新与补偿线程锁顺序不一致导致线程互相等待

## 一、这个案例怎么和你的简历对上

这个案例最适合挂在你的：

- `司库信息系统`
- `AsyncJobCenter`
- 回执处理、状态流转、失败补偿

一句话背景：

> 我们在处理任务状态更新和回执补偿时，两个线程对资源的加锁顺序不一致，结果在并发情况下形成环形等待，应用线程没有报业务异常，但接口卡住、任务处理停滞，最后通过线程 dump 定位到死锁。

## 二、出现过程

- 某类任务处理突然卡住
- 日志没有明显异常栈
- 线程池活跃线程不低，但吞吐几乎掉到 0
- 上游看到任务一直停留在“处理中”

## 三、排查过程

### 第 1 步：先判断是不是线程池打满

先看线程池，发现不是简单线程数不够，而是有一批线程长期不释放。

### 第 2 步：导线程栈

```bash
jstack <pid>
```

### 第 3 步：看锁等待关系

在线程 dump 里看到：

- 线程 A 持有 `taskLock`，等待 `receiptLock`
- 线程 B 持有 `receiptLock`，等待 `taskLock`

有时候 `jstack` 会直接提示：`Found one Java-level deadlock`

### 第 4 步：回到代码确认触发路径

#### 路径 1：正常任务执行线程
1. 先锁任务对象
2. 更新任务状态
3. 再锁回执对象

#### 路径 2：补偿线程
1. 先锁回执对象
2. 再锁任务对象

两边顺序完全相反。

## 四、根因

根因就是：

- 多线程访问多个共享资源
- 不同调用路径加锁顺序不一致
- 形成环形等待

## 五、解决过程

### 1. 先止血

- 重启卡死实例
- 暂时摘掉高风险补偿任务

### 2. 根因修复

- 统一加锁顺序
- 缩小锁范围
- 部分场景改成 `tryLock + timeout`
- 尽量避免双锁嵌套

## 六、沉淀过程

1. 统一了锁顺序规范
2. 固化了线程 dump 排查 SOP
3. 压测补了执行线程与补偿线程撞锁场景
4. 评审时检查多资源加锁顺序

## 七、1 分钟面试回答版

> 我遇到过一次比较典型的死锁问题，场景是在任务状态更新和回执补偿处理里。正常执行线程是先锁任务再锁回执，补偿线程反过来先锁回执再锁任务，高峰期并发一上来就形成了环形等待。这个问题的特点是系统没明显报错，但线程一直卡住，吞吐掉得很厉害。排查时我先看线程池，发现不是简单打满，而是有线程长时间不释放；后面导了 `jstack`，直接在 dump 里看到线程互相等待两把锁，确认是 Java 级死锁。修复上我们先重启实例止血，代码层统一了锁顺序，并把部分双锁逻辑改成 `tryLock + timeout` 和更细粒度的状态更新。后面还把多资源加锁顺序写进了评审规范。
