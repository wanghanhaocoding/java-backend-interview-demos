# 死锁案例：xtimer 执行回调线程与停用定时器线程锁顺序不一致导致互相等待

## 一、这个案例怎么和你的简历对上

这个案例直接挂你第一个项目：

- `ScheduleCenter 定时调度中心`
- 真实代码：`bitstorm-svr-xtimer`
- 真实资源：`timer_task`、`xtimer`

一句话背景：

> xtimer 在执行定时回调时，需要根据 `timer_task(timer_id, run_timer)` 找到任务，再判断 `xtimer` 主记录是否仍然启用；而管理侧如果在同一时刻停用某个定时器，又会先更新 `xtimer`，再去处理关联的 `timer_task`。如果两条路径的锁顺序相反，就可能在高峰期形成环形等待。

## 二、出现过程

- 某批定时任务处理突然卡住
- 实例没有明显业务异常栈
- 线程池活跃线程不低，但吞吐几乎掉到 0
- 管理侧出现“停用定时器”操作卡住，执行侧回调也卡住

## 三、排查过程

### 第 1 步：先判断是不是线程池打满

先看线程池，发现不是简单线程数不够，而是有几条线程长期不释放。

### 第 2 步：导线程栈

```bash
jstack <pid>
```

### 第 3 步：看锁等待关系

在线程 dump 里会看到：

- 线程 A 持有 `timer_task` 对应锁，等待 `xtimer`
- 线程 B 持有 `xtimer` 对应锁，等待 `timer_task`

有时候 `jstack` 会直接提示：

`Found one Java-level deadlock`

### 第 4 步：回到 xtimer 代码确认触发路径

#### 路径 1：执行回调线程

1. 先锁 `timer_task`
2. 更新执行状态
3. 再锁 `xtimer`，确认定时器仍处于启用态

#### 路径 2：停用定时器线程

1. 先锁 `xtimer`
2. 变更定时器状态
3. 再锁关联的 `timer_task` 做取消或补偿处理

两边顺序完全相反。

## 四、根因

根因就是：

- 多线程访问 `timer_task` 和 `xtimer` 两类共享资源
- 不同调用路径加锁顺序不一致
- 在高并发下形成环形等待

## 五、解决过程

### 1. 先止血

- 重启卡死实例
- 暂时摘掉高风险停用操作或批量补偿动作

### 2. 根因修复

- 统一 `xtimer -> timer_task` 或 `timer_task -> xtimer` 的全局锁顺序
- 缩小锁范围，不要在持锁期间做额外网络调用
- 部分场景改成 `tryLock + timeout`
- 把停用逻辑和执行后置处理拆成更细粒度动作

## 六、沉淀过程

1. 统一了 `xtimer / timer_task` 的锁顺序规范
2. 固化了线程 dump 排查 SOP
3. 压测补了“停用定时器 + 执行回调并发”场景
4. 评审时专项检查多资源加锁顺序

## 七、1 分钟面试回答版

> 我在 ScheduleCenter 里准备过一个比较典型的死锁案例，场景是在 xtimer 执行回调和停用定时器并发发生时。执行线程通常会先处理 `timer_task` 再检查 `xtimer` 主记录，而管理侧停用定时器会先改 `xtimer`，再去处理关联的 `timer_task`。如果两边锁顺序不一致，就可能形成环形等待。这个问题的特点是系统不一定直接报错，但线程会一直卡住，吞吐掉得很厉害。排查时我先看线程池，确认不是简单打满，然后导 `jstack`，直接看到两条线程分别持有 `timer_task` 和 `xtimer` 锁并互相等待，确认是 Java 级死锁。修复上先重启实例止血，代码层统一锁顺序，并把部分双锁逻辑改成 `tryLock + timeout` 和更细粒度的状态处理。
