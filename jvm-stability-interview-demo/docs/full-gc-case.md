# Full GC 案例：ScheduleCenter 批量预取过大导致老年代压力持续升高

## 一、这个案例怎么和你的简历对上

这个案例最适合挂在你的：

- `ScheduleCenter 定时调度中心`
- 高精度触发
- 时间窗口预取
- MySQL + Redis 二级存储

一句话背景：

> 我们在定时调度中心里，为了保证秒级精度，做了未来时间窗任务预取和本地缓冲，但早期批量拉取量过大、对象在队列里停留时间过长，导致老年代占用持续升高，系统虽然没立刻 OOM，但出现频繁 Full GC，调度延迟和 RT 抖动明显。

## 二、出现过程

- 每到整点、半点，大量任务集中触发
- 调度线程吞吐下降
- 回调耗时上升
- 实例不一定挂，但系统明显卡顿

监控上能看到：

- Full GC 次数明显增加
- Full GC 停顿时间拉长
- Old 区占用高且回落幅度差

## 三、排查过程

### 第 1 步：先排除线程池和下游问题

- 看线程池队列和活动线程数
- 看 Redis、MySQL、回调下游 RT

结论是：下游有波动，但不足以解释整机卡顿。

### 第 2 步：看 GC 指标

常见命令：

```bash
jstat -gcutil <pid> 1000 20
jcmd <pid> GC.heap_info
```

观察到：

- Old 区占用持续偏高
- Minor GC 后很多对象仍然存活
- Full GC 频率比正常高很多

### 第 3 步：回到对象流转链路

触发链路大概是：

1. 从 MySQL 扫描未来时间窗任务
2. 组装任务对象
3. 放入 Redis 热数据
4. 同时进本地队列等待触发
5. 触发后再构造回调报文

问题在于：

- 单批拉取太大
- 本地队列排队时间变长
- 一个任务对象在多个环节都存在副本

## 四、根因

真正根因是：

- 时间窗预取批次过大
- 本地缓冲队列没有有效背压
- 对象在扫描、预取、排队、回调构造链路里停留过久
- 大对象和中间对象被反复复制并晋升到老年代

## 五、解决过程

### 1. 先止血

- 临时下调单批预取量
- 降低同一时间窗并发触发阈值
- 扩实例分摊压力

### 2. 根因治理

- 缩小单批扫描/预取大小
- 加本地队列背压
- 热路径对象瘦身，大 payload 延后加载
- 扫描、触发、回调用不同线程池隔离
- JVM 参数调优只作为辅助

## 六、沉淀过程

1. 建了 GC 监控看板
2. 做了批次大小与堆占用的压测基线
3. 规定批处理必须有上限、队列必须有背压
4. 对 Full GC 次数和调度延迟补了告警

## 七、1 分钟面试回答版

> 我在 ScheduleCenter 里遇到过一次比较典型的 Full GC 问题。我们为了保证秒级调度精度，会提前预取未来时间窗里的任务，放到 Redis 热数据和本地缓冲里。早期某版为了追求吞吐，把单批预取量做得比较大，结果高峰期消费跟不上时，大量任务对象、JSON 和回调上下文会在本地队列里停留很久，很多对象跨过多次 Minor GC 晋升到老年代，最终导致 Full GC 非常频繁。这个问题的特点是实例没立刻挂，但 RT 和调度延迟明显抖动。排查时我先排除了线程池和下游服务瓶颈，再通过 GC 指标确认 old 区长期偏高，最后回到链路上发现是批量预取过大和背压不足。处理上我们先下调批次、扩容止血，后面把扫描批次、队列上限、对象瘦身和线程池隔离都做了治理，同时补了 Full GC 监控和压测基线。
